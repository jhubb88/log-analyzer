## 1. TL;DR

The AI Log Analyzer accepts a pasted or uploaded log file and returns a structured analysis — log type classification, severity rating, summary, timeline, errors, warnings, recurring patterns, and recommended fixes — generated by Claude. The notable parts are the synchronous flow (no queue, no polling — the user posts once and the analysis renders in the same round-trip) and the pure-stdlib Lambda that calls the Anthropic API directly via `urllib.request`, no SDK, no transitive dependencies, 1,697-byte deployment package. Stack: vanilla HTML/JS frontend on CloudFront + S3, Python 3.12 Lambda behind API Gateway, Anthropic `claude-haiku-4-5-20251001`.

## 3. Architecture

The frontend is a single-page `index.html` served from S3 through CloudFront with custom domain `log-analyzer.jimmyhubbard2.cc`. The page renders an upload UI with a drag-and-drop zone, a "Load Demo" button that fetches the bundled `demo.log`, and a paste area as a backup input. The frontend posts `{logContent, fileName}` to `POST /analyze` on API Gateway. API Gateway requires `x-api-key` and rate-limits via usage plan `iqy9wx` (2 rps, 5 burst, 100 per day).

The Lambda receives the request, validates the body, server-side-truncates the log content to 50,000 characters if exceeded, builds a structured prompt asking for a single JSON object with nine specific fields (logType, severity, summary, timeline, errors, warnings, patterns, recommendedActions, preventionTips), and calls `https://api.anthropic.com/v1/messages` directly via `urllib.request.Request`. No boto3, no `anthropic` SDK — just the Python stdlib. Headers: `x-api-key` (the Anthropic key, loaded from environment), `anthropic-version: 2023-06-01`, `content-type: application/json`. Model: `claude-haiku-4-5-20251001`, max_tokens 2000, temperature 0.1.

The Anthropic response is parsed inline, any code-fence wrapping is stripped defensively, the inner JSON is parsed, and the analysis is returned in the same HTTP round-trip. A full call typically completes in 1-3 seconds; urllib timeout is 25 seconds, API Gateway integration timeout is 29 seconds. No async self-invoke, no S3 status or result objects, no polling. The frontend renders the analysis immediately upon 200.

## 4. Current state

### Works end-to-end

- Paste or upload a log file (`.log`, `.txt`, `.csv`, `.json` up to 50,000 characters). Get back a structured JSON analysis: logType, severity, summary, timeline, errors, warnings, patterns, recommendedActions, preventionTips. Verified live 2026-05-12.
- "Load Demo" button fetches the bundled `demo.log` (~4 KB sample web-server log) so recruiters can see the analysis without finding their own log file.
- CI auto-deploys on every push to `main`. S3 sync runs unconditionally; Lambda rebuild gated by a diff-guard checking `lambda_function.py` against `HEAD^..HEAD`.
- CORS preflight against `POST /analyze` returns `Access-Control-Allow-Headers: Content-Type,x-api-key,Authorization`. The CI workflow fails the build if `x-api-key` is missing from the response.
- Single-file 1,697-byte Lambda deployment package. No pip install in CI, no transitive dependencies, no supply-chain attack surface.

### Open items

- API key embedded as a plaintext literal in `index.html` line 579. Visible to anyone running view-source on the live URL. Tracked as Future Enhancement E5.
- Frontend calls the raw `execute-api.amazonaws.com` URL directly. No CloudFront proxy path exists for `/api/*`. Tracked as E1.
- No CloudWatch alarms exist for this project. Tracked as E2.
- `portfolio-user` CLI principal lacks `cloudwatch:GetMetricStatistics` and `ce:GetCostAndUsage` permissions. 30-day invocation/error counts and per-project cost are not queryable from the CLI. Tracked as E3.
- Frontend bucket has no lifecycle configuration. Noncurrent versions accumulate indefinitely (low impact at this traffic level; not promoted to its own enhancement entry).

### Phase 4 — CI auto-deploy with smoke test, and the meta-learning moment that followed (2026-05-12)

**The smoke test** (`63e145f`, `6218f76`). Added Lambda auto-deploy to the GitHub Actions workflow, gated by a diff-guard on `lambda_function.py`. Bundled a CORS smoke test that runs after every Lambda deploy: OPTIONS request against `/analyze`, exits non-zero if `Access-Control-Allow-Headers` does not contain `x-api-key`. All actions bumped to v6 majors (Node 24).

**The test-the-smoke-test sub-arc** (`7d876db` → `4953409` → `12d4b0c`). During CI verification, we deliberately broke CORS to confirm the new smoke test would catch a regression. It did — the build failed in 30 seconds. But the `sed` substitution used for the test break over-matched and corrupted an unrelated header. A revert plus a follow-on fix restored production within minutes. The incident produced two persistent rules added to global tooling guidelines:

> **Don't Break Production to Test Verification.** Verification of a guard is the original bug being caught, not a second induced bug. The original CORS fix from `56c779f` was already proof that the bug class existed.

> **Line-Anchored Edits for Production Code.** `sed` silently over-matches. Default to Edit-with-context. `sed` is only safe when the pattern is verified unique (`grep -c` returns 1) AND the change is reversible without redeploy.

Both rules apply across the portfolio. They have prevented recurrence in resume-matcher's CI overhaul, FieldIQ's SP+ widget refactor, and ongoing commits.

### Phase 5 — TLS audit (2026-05-14)

Audited reported TLS 1.0 on the REST API, traced it to AWS-managed edge enforcement, confirmed live floor is TLS 1.2+. No action required; raw execute-api closure tracked as E1. Identical audit conclusion as resume-matcher Phase 6 — both APIs share the same EDGE-without-custom-domain architecture.

## 7. Key decisions

### 1. Pure `urllib`, no SDK

The deployment package is 1,697 bytes — a single file, four stdlib imports, zero PyPI dependencies. The resume-matcher Lambda by contrast ships a 7.7 MB, 1,961-file package with 17 pinned transitive dependencies that CI rebuilds on every backend change. With pure `urllib`, the workflow runs `zip -j lambda.zip lambda_function.py` and uploads — no `pip install -r requirements.txt -t build/` step, no transitive-version drift, no supply-chain risk surface. Three real consequences: CI deploys complete in seconds instead of ~90 seconds, every dep upgrade in the world is a non-event for this Lambda, and the diff-guard only needs to watch one file. Trade-off: hand-coded request shape means a future Anthropic API breaking change requires a manual fix, not a pip upgrade. Acceptable for a single endpoint with one prompt.

### 2. Synchronous flow over async polling pattern

Log analysis with `claude-haiku-4-5-20251001` typically completes in 1-3 seconds. Haiku is fast, the prompt is small, and there is no PDF parsing in the request path. urllib has a 25-second timeout; API Gateway integration timeout is 29 seconds; comfortable margin. The 50,000-character input cap keeps prompt size bounded; Haiku's latency stays in the 1-3 second range across the full input range. No realistic input pushes the call near the 25-second urllib timeout. The frontend posts once, blocks on the response, renders the analysis. Compare resume-matcher: `claude-sonnet-4-6` over a full resume + JD reasoning prompt typically takes 5-15 seconds — too close to the 29-second API Gateway limit to risk synchronous. Resume-matcher needs the async self-invoke + status/result pattern; log-analyzer does not. The simpler architecture has fewer failure modes: no `lambda:InvokeFunction` IAM permission, no results bucket, no PII lifecycle policy to maintain, no polling logic on the frontend, no race condition window between `status.json` and `result.json`.

### 3. OAC over OAI for the static-site bucket

The frontend bucket is fronted by Origin Access Control rather than the older Origin Access Identity pattern. Bucket policy scoped to `cloudfront.amazonaws.com` service principal with a `SourceArn` condition pinned to distribution `E3Q8ZCVRAS854T`; CloudFront can read, nothing else can. All four Public Access Block settings on. Static website hosting disabled (S3 REST endpoint via OAC). SPA-style routing handled at CloudFront with `CustomErrorResponses` mapping 403 and 404 to `/index.html` with a 200 response. OAC is the AWS-recommended pattern since 2022: SigV4 signing, support for SSE-KMS bucket encryption (not used here but available), and a more auditable trust model than OAI's anonymous-reader-by-CF-only convention. Migrated 2026-05-07 as part of the portfolio-wide OAC standardization across all eight project distributions.

### 4. TLS hardening — accepted AWS-managed edge enforcement (2026-05-14)

The `securityPolicy: TLS_1_0` reported by `get-rest-api` looked alarming at first read. EDGE-optimized REST APIs without a custom domain route through an AWS-managed CloudFront layer, and the `securityPolicy` field on the REST API itself is informational unless bound to a custom domain name. No custom domain exists for this API (`aws apigateway get-domain-names` returned `[]`), so the field is vestigial and the live TLS floor for traffic to `https://90rplm0obh.execute-api.us-east-1.amazonaws.com` is set by AWS at TLS 1.2+. No action taken at the API Gateway layer. The meaningful follow-up is closing the raw execute-api endpoint and forcing all traffic through CloudFront; tracked as Future Enhancement E1.

## 8. Future enhancements

### E1 — Close the raw execute-api endpoint (priority: medium)

The frontend currently posts directly to `https://90rplm0obh.execute-api.us-east-1.amazonaws.com/prod/analyze`. This works but bypasses CloudFront entirely: no edge caching, no WAF attachment point, no single TLS chokepoint. Two paths forward:

- **Path A — CloudFront `/api/*` behavior.** Add a second origin to distribution `E3Q8ZCVRAS854T` pointing at `90rplm0obh.execute-api.us-east-1.amazonaws.com`. Add a cache behavior with path pattern `/api/*` targeting the new origin, `AllowedMethods` including POST + OPTIONS, no caching. Rewrite `API_ENDPOINT` in `index.html` line 517 from the raw URL to `'/api/analyze'`. Update the CORS smoke test in `.github/workflows/deploy.yml` line 63 to hit the CloudFront-proxied path.
- **Path B — API Gateway custom domain.** Provision `api.log-analyzer.jimmyhubbard2.cc` as a custom domain, attach ACM cert, map to API `90rplm0obh` stage `prod`. Set TLS policy `TLS_1_2`. Rewrite `API_ENDPOINT` in `index.html` line 517 to the new custom domain.

Either path eliminates the raw endpoint as a network reach surface. After either lands, set `disableExecuteApiEndpoint=true` and redeploy the prod stage.

### E2 — CloudWatch alarms for errors and 5xx (priority: medium)

Zero alarms exist for this project right now. Suggested baseline:

- Lambda `Errors >= 3 in 5 min` on the `jimmy-log-analyzer` function (catches sustained backend failures: Anthropic API outage, urllib timeout, panic in the handler).
- API Gateway `5XXError >= 5 in 5 min` on the `90rplm0obh/prod` stage (catches integration failures and Lambda invocation errors that surface as 502/504).
- Anthropic API failures show up as Lambda errors (urllib raises on HTTPError or timeout), so the Lambda alarm covers that path.

SNS topic with email subscription for notifications. One-time setup, low ongoing maintenance.

### E3 — Grant `portfolio-user` CLI observability reads (priority: low)

`cloudwatch:GetMetricStatistics` and `ce:GetCostAndUsage` both return AccessDenied for the `portfolio-user` IAM principal. Today this means scripted observability reads — invocation counts, error rates, per-project cost — fall back to the AWS Console. Acceptable for now (no daily ops burden), but blocks any future scripted reporting. Grant requires either inline policy addition or attached managed policies (`CloudWatchReadOnlyAccess` plus a Cost Explorer policy). Applies portfolio-wide.

### E4 — End-to-end automated test in CI (priority: medium)

CI today verifies CORS preflight but not the full POST → 200 → parsed-analysis loop. A synthetic log file + a real POST + a check on the response shape (specifically, a 200 status and the presence of expected fields like `logType` and `severity`) would catch regressions in Lambda integration, Anthropic API breakages, and JSON-parse errors — all silent failure modes that the CORS smoke test cannot reach. Cost: one Haiku call per CI run, negligible at portfolio commit frequency.

### E5 — Move API key out of client-side code (priority: medium)

The `x-api-key` value is embedded as a plaintext literal in `index.html` line 579 — visible via view-source on the live URL. The 100/day usage plan quota limits damage. Proper fix: route through a CloudFront function or Lambda@Edge that injects the key server-side, OR drop API key auth in favor of WAF + rate limiting.
